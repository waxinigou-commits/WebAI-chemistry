# Feature Flow

这是一个最小可恢复的串行执行器，用来把“连续做 5 个功能，每完成一个就进入下一个”落成真实机制。

## 目标

解决以下问题：
- 连续执行中途超时或 SIGKILL 后无法知道跑到哪里
- preview 端口残留导致后续测试互相污染
- 单条长指令执行到一半就停，无法续跑
- 每一步完成后没有持久化状态

## 可用命令

```bash
npm run flow:reset
npm run flow:run
npm run flow:status
```

## 状态文件

执行状态保存在：

```bash
.flow/feature-flow-state.json
```

字段含义：
- `status`: `idle | running | blocked | completed`
- `currentIndex`: 当前执行到的 feature 下标
- `maxRetries`: 环境类失败的自动重试次数
- `features[]`: 每个功能的执行状态、尝试次数、退出码、信号、错误摘要
- `history[]`: 最近的流程事件历史

## 默认串行顺序

1. 功能 1 火焰开关
2. 功能 2 热区判定
3. 功能 3 重置逻辑
4. 功能 4 状态面板一致性
5. 功能 5 总体验收

## 执行规则

每个功能都按下面流程执行：

1. 标记为 `running`
2. 清理 4176-4185 端口上的残留 preview 进程
3. 运行对应测试命令
4. 成功则标记为 `passed` 并自动进入下一个
5. 失败则判断是否属于可重试环境错误
6. 如果可重试，则自动再跑，最多 `maxRetries + 1` 次
7. 如果仍失败，则流程进入 `blocked`

## 当前定位

这个 flow 先解决“连续执行机制缺失”的问题，
不等于已经解决具体功能 bug。

也就是说：
- 它能保证流程知道自己跑到哪一步
- 能自动清理环境并做有限重试
- 但如果第 4 个功能本身逻辑没修好，flow 还是会停在第 4 步并留下明确状态

## 下一步建议

后续可以继续补两个方向：

1. `reporter`：每步完成后自动生成一段适合群里播报的短消息
2. `repair branch`：当某步测试失败时，自动进入修复脚本，而不是只停在 blocked
